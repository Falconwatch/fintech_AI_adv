# Domain Model Training

## Цель

Проект решает задачу `Home Credit Default Risk` через набор доменных моделей.

На текущем этапе реализован первый уровень системы:

- для каждого домена строится отдельная модель
- каждая модель обучается независимо
- все доменные модели используют единое разбиение исходного train на `development` и `holdout`

Идея второго уровня уже заложена в архитектуру: в дальнейшем предсказания доменных моделей на `holdout` будут использоваться как признаки для метамодели.

## Домены

Сейчас в проекте предусмотрены следующие домены:

- `application`
- `bureau`
- `previous_application`
- `installments`
- `pos_cash`
- `credit_card`

Для каждого домена формируется таблица вида:

`SK_ID_CURR -> domain_features`

## Источники данных

Основной таргет и идентификаторы берутся из:

- `application_train.csv`
- `application_test.csv`

Дополнительные доменные признаки строятся из соответствующих таблиц соревнования Kaggle.

## Общее разбиение train

Перед обучением моделей исходный `application_train` один раз делится на две части:

- `development`
- `holdout`

Свойства этого разбиения:

- разбиение общее для всех доменных моделей
- разбиение стратифицировано по `TARGET`
- разбиение сохраняется на диск и затем переиспользуется

Это нужно для того, чтобы все доменные модели обучались в одинаковых условиях, а `holdout` оставался чистой отложенной выборкой.

## Построение доменных признаков

### Application

Для домена `application` используются признаки напрямую из:

- `application_train.csv`
- `application_test.csv`

Колонка `TARGET` удаляется из признаков.

### Остальные домены

Для таблиц с несколькими строками на клиента выполняется агрегация по `SK_ID_CURR`.

Используемая логика:

- числовые признаки агрегируются статистиками `count`, `mean`, `min`, `max`, `sum`, `std`
- категориальные признаки кодируются через `get_dummies`
- по dummy-признакам считаются агрегаты `mean` и `sum`

Дополнительно:

- для `bureau` сначала агрегируется `bureau_balance` по `SK_ID_BUREAU`, после чего результат присоединяется к `bureau`
- для `installments` вычисляются производные признаки:
  - `payment_diff = AMT_PAYMENT - AMT_INSTALMENT`
  - `days_diff = DAYS_ENTRY_PAYMENT - DAYS_INSTALMENT`

После агрегации доменные таблицы присоединяются к спискам клиентов из `application_train` и `application_test`, чтобы получить полный train/test по каждому домену.

## Обучение доменной модели

Для каждого домена выполняется следующий pipeline:

1. Загружается доменный датасет.
2. Доменные признаки объединяются с `TARGET`.
3. Строки делятся на `development` и `holdout` по заранее сохранённому split.
4. Внутри `development` выполняется подбор гиперпараметров.
5. Лучшая конфигурация переобучается на всём `development`.
6. Финальная оценка качества проводится только на `holdout`.

Модель первого уровня:

- `LightGBM`
- задача: бинарная классификация
- метрика: `ROC-AUC`

## Кросс-валидация внутри development

Чтобы улучшить качество доменных моделей и не использовать `holdout` для подбора параметров, внутри `development` реализована отдельная стратифицированная кросс-валидация.

Схема:

- `development` разбивается на `K` фолдов
- на каждом шаге модель обучается на `K-1` фолдах
- качество считается на оставшемся фолде
- для каждой конфигурации гиперпараметров считается средний `ROC-AUC` по всем фолдам

`holdout` в этом процессе не участвует.

## Перебор гиперпараметров

Сейчас используется компактный grid search по нескольким ключевым параметрам `LightGBM`.

В текущей версии перебираются:

- `learning_rate`
- `num_leaves`
- `min_data_in_leaf`
- `feature_fraction`
- `bagging_fraction`

Для каждой конфигурации сохраняются:

- средний `ROC-AUC` по CV
- стандартное отклонение `ROC-AUC`
- средний лучший `best_iteration`

После этого выбирается конфигурация с максимальным средним `ROC-AUC`.

## Финальное обучение домена

После выбора лучших гиперпараметров:

- модель обучается на всём `development`
- используется early stopping на `holdout`
- затем формируются предсказания на:
  - `development`
  - `holdout`
  - `application_test`

## Сохраняемые артефакты

Для каждого домена сохраняются:

- сериализованная модель
- метрики
- метаданные
- таблица `feature_importance`
- таблица `cv_results`
- предсказания на `development`
- предсказания на `holdout`
- предсказания на `test`

## Что важно с точки зрения будущего стекинга

Текущая схема уже подготавливает систему к следующему этапу.

Ключевой принцип:

- метамодель не должна обучаться на предсказаниях, полученных на тех объектах, на которых соответствующая доменная модель обучалась

На текущем этапе это обеспечивается так:

- доменные модели подбираются и обучаются внутри `development`
- метамодель в будущем сможет использовать предсказания доменных моделей на `holdout`

Таким образом, `holdout` служит честным источником признаков второго уровня.
